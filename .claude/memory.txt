## API Pattern (next-ts-api)

### Setup
1. Plugin in `next.config.ts`:
```typescript
import { nextTsApi } from "next-ts-api/config";
const withNextTsApi = nextTsApi({ outDir: "types", outFile: "api.ts" });
export default withNextTsApi(nextConfig);
```

2. Types auto-generated at `types/api.ts` when dev server runs

3. API client at `lib/api/client.ts`:
```typescript
import { createNextFetchApi } from "next-ts-api";
import { type ApiRoutes } from "@/types/api";
export const api = createNextFetchApi<ApiRoutes>();
```

### Query Structure
```
lib/
├── api/
│   ├── client.ts        # api = createNextFetchApi<ApiRoutes>()
│   ├── key-generator.ts # createKey() for query keys
│   └── index.ts
└── queries/
    ├── conversations.ts # conversationsQuery(authKey) factory
    ├── listings.ts      # listingsQuery(authKey) factory
    ├── user.ts          # userQuery(authKey) factory
    └── index.ts
```

### Usage
```typescript
import { useQuery } from "@tanstack/react-query";
import { conversationsQuery } from "@/lib/queries";

const authKey = user?.id;
const { data } = useQuery(conversationsQuery(authKey).list());
```

### Key Features
- Types auto-generated by next-ts-api plugin
- Query factories accept authKey for cache invalidation per user
- createKey() flattens objects into query key arrays

### Important Notes
- Queries: Use query factories (conversationsQuery, listingsQuery, userQuery)
- Mutations: Use api() client for all JSON requests (never use direct fetch)
- Query factories need return types for proper inference (e.g., ListingsResponse)
- Components should derive types from query data, avoid explicit interface declarations
- For FormData uploads (images, avatars), direct fetch() is acceptable
- For query params in URLs, use `query` option: `api("route", { query: { key: value } })`
- Error responses need type assertion: `await res.json() as { error?: string }`
- Fixed bug: handleDeleteImage was missing `query: { imageId }` — always check route's query/param types match client call

### API Route Typing
Use `NextApiRequest<BodyType, QueryType>` from 'next-ts-api' for typed request bodies and query params:
```typescript
import { NextApiRequest } from "next-ts-api";

// POST with typed body
interface LoginBody { email: string; password: string; }
export async function POST(request: NextApiRequest<LoginBody>) {
  const body = await request.json(); // typed as LoginBody
}

// GET with typed query params
interface ListingsQuery { page?: string; limit?: string; search?: string; }
export async function GET(request: NextApiRequest<unknown, ListingsQuery>) {
  const page = request.nextUrl.searchParams.get("page"); // typed
}

// Dynamic route with params
export async function GET(
  request: NextApiRequest<unknown>,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;
}
```

Note: Use `unknown` (not `never`) for routes without body/query to maintain Next.js compatibility.

### tsconfig.json
Next.js adds `.next/types/**/*.ts` to includes. Add `.next` to excludes to avoid validator conflicts:
```json
"exclude": ["node_modules", ".next"]
```
